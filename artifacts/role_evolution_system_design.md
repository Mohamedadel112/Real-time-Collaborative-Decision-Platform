# User Weight & Role Evolution System
**Production-Ready Architecture Document**

This document outlines the design and implementation of an enterprise-grade "User Weight & Role Evolution System" for a real-time collaborative decision-making platform built on **NestJS, Prisma, and Redis**.

---

## 1. System Overview

**Purpose:**  
The purpose of this system is to dynamically assign a calculated voting weight to users based on their active performance, historical accuracy, and trusted status, while facilitating organic transitions through platform roles. Simple polling is inadequate for complex issues; this system relies on meritocratic consensus to ensure high-quality outcomes.

**Role Distinctions:**
- **User:** The baseline. Participates normally.
- **Trusted User:** A user specifically invited by an Admin, representing off-platform credentials or proven trust. This user enjoys a "warm start" (higher initial baseline weight).
- **Expert:** A dynamic, temporary state achieved strictly by quantifiable performance in specific domains. It grants massive weight multipliers.
- **Admin:** System maintainer with absolute platform authority but standard voting weight.

**Dynamic vs Static Roles:**  
Static roles permanently anchor influence, inevitably leading to decay in quality if the user stops participating meaningfully. Dynamic roles (like the Expert) must be continuously *earned* and automatically *downgraded* upon poor performance or inactivity, ensuring the platform's consensus is always driven by the sharpest minds.

---

## 2. Role Hierarchy & Evolution 

Users transition through a continuous funnel:  
`Guest → User → Trusted User → Expert → Admin`

*   **Guest → User:** Organic transition upon email verification and KYC/registration.
*   **User → Trusted User:** Requires an `inviteToken` generated by an `Admin`. 
*   **User/Trusted User → Expert:** Controlled algorithmically. Evaluated via cron jobs, event-listeners, and triggers dynamically observing `accuracyScore` and `reputation`. 

*Note: The "Expert" is not a fixed database column (`role: 'EXPERT'`), but rather a computed state mapped to a specific Domain/Skill. A user might be an `Expert` in `FINANCE` but a standard `User` in `DESIGN`.*

---

## 3. Trusted User System

Trusted Users enter the ecosystem via administrative decree. Because they are often industry professionals sourced externally, they require a higher baseline vote weight starting from Day 1 to immediately impact consensus safely.

**Database Concept:**
```graphql
model User {
  id               String
  role             Role      // 'USER' | 'ADMIN'
  isInvitedByAdmin Boolean   @default(false)
  invitedById      String?   // Tracks which Admin issued the invite
}
```

**Risk Management (Admin Abuse):**
- **Invite Limits:** Admins are functionally rate-limited depending on their tenure (e.g. 5 invites globally per week per admin).
- **Audit Logs:** Every `isInvitedByAdmin = true` transition triggers an immutable row in the `AuditLog` table.
- **Trust Decay:** If a Trusted User's `accuracyScore` falls below 40%, the system automatically flags them for downgrade, stripping the "Trusted" baseline modifier regardless of Admin invitation.

---

## 4. Expert System (Dynamic Role)

Expertise is an algorithmic destination, not a button an Admin presses. 

**Conditions for Expertise:**
1. **Participation (`votesCount`):** Minimum of 50 votes cast in a specific domain.
2. **Accuracy (`accuracyScore`):** Must maintain a win-rate > 75% alongside the consensus or ultimate truth over a rolling 30-day window.
3. **Reputation (`reputation`):** Must exceed the 90th percentile of the platform.

```typescript
function checkIfExpert(userSkill: UserSkill): boolean {
  if (userSkill.totalVotes < 50) return false;
  if (userSkill.accuracyScore < 75.0) return false;
  if (userSkill.reputation < 500) return false; // Minimal threshold
  return true;
}
```

**Downgrade Logic (Dynamic Removal):**  
A nightly NestJS `@Cron()` task iterates over all `UserSkill` records marked as `isExpert: true`. If `checkIfExpert()` returns false (due to sudden poor accuracy or inactivity-based reputation decay), the system flips `isExpert` back to `false` automatically.

---

## 5. Skill-Based Expertise

An individual is rarely an expert at *everything*. Weights must be domain-isolated.

**Data Segregation:**
Decisions are tagged at creation (e.g., `tag: 'ENGINEERING'`). The weight calculation searches the user's `UserSkill` array for a matching tag.

**UserSkill Database Approach:**
```prisma
model UserSkill {
  id          String  @id @default(uuid())
  userId      String
  skillTag    String
  reputation  Float   @default(0)
  accuracy    Float   @default(0)
  totalVotes  Int     @default(0)
  isExpert    Boolean @default(false) // Dynamic algorithmic flag

  user        User    @relation(fields: [userId], references: [id])
  @@unique([userId, skillTag])
}
```

If a User is an Expert in `ENGINEERING`, their vote counts as $5.0x$. If the same User votes on a `MEDICINE` decision, they default to their standard weight.

## 6. Weight Calculation Engine

The mathematics powering the platform. Weight is computed real-time dynamically at the moment of voting. It acts cumulatively across specific multipliers.

```typescript
@Injectable()
export class WeightEngineService {
  calculateWeight(user: User, userSkill: UserSkill | null, decision: Decision): number {
    let weight = 1.0; // Base Weight
    const breakdown = [];

    // 1. Warm Start / Trusted User
    if (user.isInvitedByAdmin) {
      weight += 2.0; 
      breakdown.push('+2.0 Trusted Baseline');
    }

    // 2. Skill Match & Expertise
    if (userSkill && userSkill.skillTag === decision.tag) {
      weight += 1.0; 
      breakdown.push('+1.0 Skill Match');
      
      if (userSkill.isExpert) {
        weight *= 1.5;
        breakdown.push('*1.5 Domain Expert Multiplier');
      }
    }

    // 3. Reputation Scaling (Logarithmic)
    const effectiveRep = userSkill ? userSkill.reputation : user.globalReputation;
    const repBonus = Math.max(0, Math.log10(Math.max(1, effectiveRep)));
    weight += repBonus;
    breakdown.push(`+${repBonus.toFixed(2)} Reputation Bonus`);

    return { 
      totalWeight: Number(weight.toFixed(3)), 
      explainability: breakdown 
    };
  }
}
```

## 7. Real-Time Integration (Architecture)

1. **User Votes:** The client emits a `castVote` payload via WebSockets to the NestJS `@WebSocketGateway()`.
2. **Lock Acquisition:** The `VoteGateway` invokes Redis: `SET lock:vote:${userId} NX PX 2000`. If it fails, the socket emits an error (rate-limited / duplicate).
3. **Weight Calculated:** The user profile and `UserSkill` are pulled from a Redis Cache (`GET user:${userId}`), and fed into `WeightEngineService`.
4. **Transaction:** Prisma processes the `$transaction()` writing the vote, generating the explanation, and updating the Decision total weight.
5. **Pub/Sub Broadcast:** NestJS emits an event via the Redis WebSocket Adapter to all clients subscribed to `room_${roomId}` with the new dynamic totals.

---

## 8. Edge Cases

- **New Users (No Rep):** The logarithmic calculation outputs `0` reputation bonus, preventing errors. They vote with exactly `1.0` base weight.
- **Trusted but Inactive Users:** The system enforces continuous validity. If an admin invites someone who goes rogue, they lose Trusted status as soon as their accuracy falls below `40%`. 
- **Fast Promotion (Sybil Gaming):** Minimum bounds (e.g., `totalVotes > 50`) prevent users from creating duplicate accounts and jumping straight to Expert on 3 lucky guesses.

---

## 9. Anti-Abuse & Security

- **Weight Caps & Normalization:** `totalWeight` computation enforces a hard ceiling (e.g. `Math.min(weight, 15.0)`). One expert shouldn't outweigh 50 regular users combined.
- **Reputation Decay (Time-Value):** A Redis scheduled worker (`XADD` streams or BullMQ) processes weekly `-2%` decay algorithms on `reputation`. Expertise requires steady upkeep.
- **Anomaly Detection:** If an `EXPERT` suddenly votes contrarian to 99% of the chamber 3 decisions in a row, a triggered alert suspends their multiplier pending automated or manual review.

---

## 10. Core Design Patterns Utilized

1. **Strategy Pattern:** Instead of a giant nested `if/else`, the `WeightEngine` uses an array of initialized strategies (`RoleStrategy`, `ExpertStrategy`, `ActivityStrategy`) that it maps over identically, decoupling the math algorithms from the host logic.
2. **Observer Pattern:** The `@OnEvent('vote.validated')` decorator allows isolated worker modules to observe when a decision finalizes, subsequently updating accuracy scores and triggering the downgrade/upgrade of the Expert `State` asynchronously.
3. **State Pattern:** The `Expert` entity behaves as a distinct mechanical State. Upon entering this State, the user gains UI badges, notification scopes, and different analytical weight pathways.

---

## 11. Gamification

- **Trust Badges:** Unlocking "Trusted User" adorns the UI with a distinct verified badge.
- **Expert Crowns:** Whenever an expert enters a chamber touching their domain, a persistent animation or distinct colored crown appears next to their avatar, communicating authority.
- **Influence Indicators:** Before a user votes, the UI fetches a "predicted weight module", motivating them to see: *"Your exact influence on this decision is 6.2x."*

---

## 12. Explainability (Deep Analytics)

Black-box algorithms are a liability. When calculating a vote's weight, the engine returns an explicit `explainability` array (as shown previously). This breakdown string is immediately persisted onto the `Vote` model using JSON metadata:

`[ "+2.0 Trusted Baseline", "+1.0 Skill Match", "+1.2 Reputation Bonus" ]`

In the Analytic UI, any user can click on a vote to see transparent, auditable factors outlining entirely *why* John Doe commanded 4.2x raw power over the process.

## 14. Nested Code Implementations

### WebSocket Gateway (Vote Execution)
```typescript
@WebSocketGateway()
export class VoteGateway {
  constructor(
    private readonly voteService: VoteService,
    @InjectRedis() private readonly redis: Redis
  ) {}

  @SubscribeMessage('castVote')
  async handleVote(client: Socket, payload: { decisionId; optionId }) {
    const userId = client.data.user.id;
    const lockKey = `lock:vote:${userId}:${payload.decisionId}`;

    const lock = await this.redis.set(lockKey, 'BUSY', 'PX', 2000, 'NX');
    if (!lock) return { error: 'Processing duplicate vote request.' };

    try {
      const result = await this.voteService.processVote(userId, payload);
      
      // Redis Pub/Sub Broadcast
      this.server.to(`room_${result.roomId}`).emit('decisionUpdate', result);
      return { success: true, weight: result.finalWeight, explain: result.breakdown };
      
    } finally {
      await this.redis.del(lockKey); // Release
    }
  }
}
```

### Expert Check Logic (Cron Job / Event Hook)
```typescript
@Injectable()
export class RoleEvolutionService {
  constructor(private prisma: PrismaService) {}

  @Cron(CronExpression.EVERY_DAY_AT_MIDNIGHT)
  async evaluateExperts() {
    const skills = await this.prisma.userSkill.findMany();
    
    for (const skill of skills) {
      // Evaluation algorithm directly dictating 'Dynamic States'
      const shouldBeExpert = skill.totalVotes >= 50 && skill.accuracyScore > 75.0 && skill.reputation > 500;

      if (skill.isExpert !== shouldBeExpert) {
        await this.prisma.userSkill.update({
          where: { id: skill.id },
          data: { isExpert: shouldBeExpert }
        });
        // Publish Event (Observer Pattern) alerting the UI the state changed
        this.eventEmitter.emit('user.expertise.changed', { userId: skill.userId, newStatus: shouldBeExpert });
      }
    }
  }
}
```

## 15. Real-World Scaling

A true production instance managing over 250,000 real-time connections relies heavily on Horizontal Scalability.
- **Instance Distribution:** Deploy exact identical stateless NestJS Docker containers scaled dynamically via Kubernetes (HPA based on CPU/RAM load).
- **Socket.io Redis Adapter:** The standard NestJS built-in engine isn't sufficient for isolated instances. When Instance A triggers a `decisionUpdate`, the `RedisIoAdapter` ensures it pushes the payload down internal Redis Pub/Sub channels to physically broadcast across Instance B, Instance C, and Instance D to all 5,000 respective users concurrently.
- **Cache Invalidation:** Ensure that updating the `isExpert` string during the midnight cron job natively clears the Redis `user:${userId}` cache; otherwise, highly active nodes will continue assigning old metrics inside the WeightService.
